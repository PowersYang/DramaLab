# 支付接入剩余工作清单

这份文档不是介绍“已经支持什么”，而是专门说明 DramaLab 当前支付能力还剩哪些事情没做完，以及建议按什么顺序继续推进。

当前代码现状：

- 已支持微信支付 `Native` 下单
- 已支持支付宝 `alipay.trade.precreate` 下单
- 已支持微信支付回调验签与密文解密
- 已支持支付宝异步通知验签
- 已支持支付成功后的幂等入账
- 已支持重复回调不重复加豆
- 已支持前端 `/studio/billing` 发起 PC 扫码充值

但这还不是“支付产品完全上线”的最终形态。下面这些事情，仍然建议继续做完。

## 一、P0：上线前必须补齐

这部分建议在正式对外开放前全部完成。

### 1. 真实商户参数与证书落地

当前代码已经支持从 `backend/.env` 读取真实配置，但还需要你们把正式商户参数准备完整：

- 微信支付：
  - `WECHAT_PAY_APP_ID`
  - `WECHAT_PAY_MCH_ID`
  - `WECHAT_PAY_MCH_SERIAL_NO`
  - `WECHAT_PAY_PRIVATE_KEY_PATH`
  - `WECHAT_PAY_API_V3_KEY`
  - `WECHAT_PAY_PLATFORM_CERT_PATH`
- 支付宝：
  - `ALIPAY_APP_ID`
  - `ALIPAY_APP_PRIVATE_KEY_PATH`
  - `ALIPAY_PUBLIC_KEY_PATH`

建议动作：

- 把证书统一放到固定目录，例如 `/opt/dramalab/certs/`
- 确认运行进程有读取权限
- 不要把私钥直接提交进仓库

验收标准：

- 后端启动后能正常读取这些配置
- 生产机上证书路径真实存在

### 2. 测试商户环境实测一轮

虽然代码已经打通，但必须用真实渠道实测。

建议至少跑这几条：

1. 微信扫码支付成功
2. 支付宝扫码支付成功
3. 同一订单重复回调
4. 用户不支付，二维码自然过期
5. 用户创建后手动取消订单

验收标准：

- `payment_orders.status` 与实际支付结果一致
- `billing_transactions` 中只产生一次充值流水
- 组织余额变化正确

### 3. 服务器公网回调链路验证

支付平台回调不是浏览器请求，很多时候本地看着能跑，公网实际回不来。

必须确认：

- `PAYMENT_NOTIFY_BASE_URL` 是支付平台能访问到的地址
- 反向代理没有拦截 `POST /billing/payment-providers/*/notify`
- HTTPS 证书正常
- 网关层不会把请求体改写

建议动作：

- 在线上域名下直接 `curl` 验证 notify 路由能到应用
- 打一条测试日志，确认真实回调已经进入 FastAPI

验收标准：

- 支付平台后台能看到回调成功
- 本地服务日志能看到 notify 请求

## 二、P1：建议尽快补齐

这部分不一定阻塞首个测试用户，但越早补越稳。

### 4. 主动查单能力

当前主链路主要依赖支付平台回调。实际线上会遇到这几种情况：

- 用户已支付，但回调延迟
- 支付平台回调失败
- 用户页面一直停在“待支付”，但后台其实已成功

建议新增：

- 微信主动查单接口封装
- 支付宝主动查单接口封装
- 一个统一的“订单状态刷新”服务

建议后续接口：

- `POST /billing/payment-orders/{order_id}/refresh`

建议行为：

- 如果订单仍是 `pending`
- 后端主动向微信/支付宝查单
- 若确认支付成功，则补触发 `mark_order_paid(...)`

验收标准：

- 回调丢失时，后台仍能通过查单补到账

### 5. 平台级支付对账任务

当前账务和支付订单已经有基础闭环，但还缺每日对账。

建议补：

- `payment_reconcile_runs`
- `payment_reconcile_items`
- 定时任务按日对账

建议检查维度：

- 支付平台成功，但本地未入账
- 本地已入账，但支付平台未成功
- 金额不一致
- 退款状态不一致

验收标准：

- 能产出每日对账结果
- 异常订单能被平台管理端看见

### 6. 平台管理端支付订单页

当前前端主入口在普通组织账本页，更偏“用户发起充值”。但运营同学需要单独的后台视角。

建议新增平台级页面：

- 全部支付订单列表
- 按状态筛选
- 按渠道筛选
- 按组织筛选
- 查看支付事件时间线

推荐路由：

- `/studio/payment-admin`

建议展示字段：

- 订单号
- 组织
- 用户
- 支付渠道
- 金额
- 到账豆数
- 状态
- 创建时间
- 支付时间
- 外部交易号

验收标准：

- 平台运营能不查数据库就定位支付问题

## 三、P2：商业化后必须做

这部分是从“能用”走向“稳态商业化”的关键。

### 7. 退款闭环

当前系统只做了充值成功入账，还没有完整退款链路。

后续建议补：

- 退款申请对象
- 第三方退款接口封装
- 退款回调或主动查退款
- 账本冲减 / 退款流水

需要特别注意：

- 如果用户已消费豆，退款规则怎么定
- 是先退人民币还是先扣回未消费豆
- 部分退款如何映射豆余额

验收标准：

- 退款成功后，现金侧和豆账侧一致

### 8. 充值套餐与赠送规则产品化

当前充值金额主要还是自由金额 + 后端赠送规则。

后续建议补成正式产品模型：

- 充值套餐表
- 套餐上下架
- 套餐展示顺序
- 首充专享
- 节日活动赠送

验收标准：

- 前端展示固定套餐
- 后台可配置
- 下单时固化套餐快照

### 9. 风控与限流

上线真实支付后，风险会明显增加。

建议增加：

- 单用户短时间创建订单限流
- 单组织充值频率限制
- 回调来源日志与异常告警
- 支付失败次数监控

验收标准：

- 异常刷单和错误回调可追踪

## 四、推荐的实施顺序

建议严格按这个顺序推进：

1. 填真实 `.env` 与证书路径
2. 在测试商户环境完成微信/支付宝真实支付联调
3. 验证公网回调
4. 补主动查单
5. 补平台支付管理页
6. 补每日对账
7. 最后再做退款和套餐产品化

原因很简单：

- 没真实联调前，不要急着做大量运营页面
- 没主动查单前，线上会因为回调偶发失败产生脏单
- 没对账前，商业化规模一上来，问题只能靠查库和人工排查

## 五、建议你接下来立刻做的事

如果你要最快把这条线推进到可上线测试，我建议下一步就做这 4 件：

1. 准备微信支付和支付宝测试商户参数
2. 把证书放到服务器固定目录
3. 填写 `backend/.env`
4. 在测试环境真实扫一笔微信和支付宝订单
